GOLC × Claude Max — Action Playbook

Nine concrete plays, ordered by when to run them. Grounded in the repo's actual state on 2026-07-20: Phase 1 executing, 6/29 plans done, waves 2–18 remaining, model_profile: "inherit", Linear mappings pending.

Do now (before the next execute wave)

NOW1 · Stop paying Fable prices for executor work

The problem, verbatim from .planning/config.json: "model_profile": "inherit". Every gsd-executor, verifier, and researcher subagent inherits your interactive session's model. With 23 plans left in Phase 1, that's ~23 executor runs plus checkers all drawing from the all-models weekly cap.

Do this:

/gsd-settings        →  Model profile  →  pick the balanced/budget profile
                        (executors on Sonnet, planners/checkers stay on the premium model)

Why it works: Max has two weekly caps — all-models and a separate Sonnet-only cap. Executor work (mechanical: follow plan, write code, commit) is exactly what Sonnet 5 is for. Moving it frees the premium cap for planning and review, roughly doubling how many plans you clear per week. Plan quality is unaffected — plans are already written by the planner before an executor sees them.

NOW2 · Let the waves actually fan out

State: "parallelization": true is already set, and each remaining plan carries wave metadata — e.g. 01-04-PLAN.md has wave: 7, depends_on: [01-16, 01-18], autonomous: true. Waves 2–18 remain. On Pro-level limits, parallel executor agents would eat a 5-hour window fast; on Max they won't.

Do this:

# Right after a 5-hour usage window resets, launch the phase executor
# and let same-wave plans run as parallel background agents:
/gsd-execute-phase 1
  • Time big waves to a fresh window: check remaining budget at claude.ai → Settings → Usage before launching.
  • Plans marked autonomous: true won't stop for you — waves can run while you're away; only checkpoint plans (like the 01-12 npm-pin gate you already cleared) need you present.
  • If two independent chunks of Phase 1 are both unblocked (config-side vs. trace-side), /gsd-workstreams + git worktrees run them without file collisions — relevant since branching_strategy: "none" means everything lands on master.

NOW3 · Connect the Linear MCP before the reconciliation plans need it

State: .planning/linear-map.json has milestone:v1 → status: "pending", linear_uuid: null. Later Phase 1 plans build the reconciliation preview (LINR-02/03/04). When they execute, having a live credential-external channel to your actual Linear workspace lets you test the preview against reality instead of mocks only.

Do this (5 minutes):

# Hosted server, OAuth in the connector — no credentials touch the repo,
# which is exactly what CONF-03 / the clean-checkout rule requires:
claude mcp add --transport http linear https://mcp.linear.app/mcp

# then in a session:
/mcp        # authenticate Linear
# smoke test: "list my Linear teams and projects"

Also decide the workspace taxonomy now (one Linear project for milestone v1? phases as projects or milestones?) — config/integrations/linear.toml declares taxonomy and env-var names only, and it's cheaper to settle that before plans 01-2x bake it in. Known caveat: 2026 reports of intermittent connector auth drops; repo-side tooling stays authoritative (the roadmap already mandates this).

WEEK4 · One scheduled job: the clean-checkout truth test

Why this job: Phase 1's core promise is "a clean checkout bootstraps offline from pins with byte-identical results." That claim rots silently as plans land. A schedule re-proves it while you sleep.

Reality check first: Anthropic's cloud Routines run on Linux — golc.ps1 is Windows PowerShell 5.1, so the real entrypoint can't run there. Two options, use A:

Option A — local scheduled task on this PC (runs the real thing):

/schedule
# prompt for the task, verbatim:
"Clone github.com/lnorton89/golc into a fresh temp directory. Run
 powershell -NoProfile -File .\golc.ps1 bootstrap, then
 .\golc.ps1 test --quick --scope config-local and
 .\golc.ps1 test --quick --scope linear-catalog.
 Confirm bootstrap made zero archive-source calls on a second run.
 Report pass/fail per step with the first failing error verbatim."
# cadence: nightly

Option B — cloud Routine (Linux, partial coverage): limit it to go build ./... + go vet ./... + module-graph checks against the pinned toolchain version read from config/toolchain.toml. Weaker, but survives your PC being off. Note cloud cron is UTC.

At the Phase 1 ship gate

GATE6 · One paid ultrareview as a calibration run

State: GSD's built-in review already runs (code_review: true, depth: "standard"). Ultrareview is the paid escalation: cloud agent fleet + independent verification pass, only confirmed bugs surface. ~$5–20 per run, billed outside the subscription.

Do this when /gsd-verify-work passes for Phase 1:

/gsd-pr-branch        # builds a clean branch with .planning/ commits filtered out
/code-review ultra    # review THAT branch — product code only, smaller diff, lower cost

Phase 1 is the cheapest possible calibration: mostly config/bootstrap Go code. If the confirmed-bug rate impresses you, budget one run per phase gate — it matters most at Phase 3 (deterministic engine), Phase 7 (API security), and Phase 8 (sandbox boundary). If it doesn't, you've spent one coffee's worth finding out.

GATE7 · Route Phase 2's research debt through ultraplan

State: the roadmap flags Phase 2 as needing deep research: canonical fixture semantics, pool propagation rules, OFL snapshot/licensing, GDTF preservation, hazardous attributes, physical validation corpus. That's days of research that would otherwise occupy your terminal and your all-models cap interactively.

Standing changes (set once)

SETUP8 · Make every agent session self-check through the pinned toolchain

The gap: subagents and future routines can edit Go files and "finish" without running your quick scopes. A Stop hook closes it mechanically — when a session that touched internal/ ends, the pinned-toolchain tests run and failures bounce back into the session.

Do this: run /update-config and ask for exactly this:

"Add a Stop hook to .claude/settings.json for this project: when a session
 modified any file under internal/ or cmd/, run
   powershell -NoProfile -File golc.ps1 test --quick --scope config-local
 and block completion on failure, feeding stderr back to the session."

As more scopes exist (a linear-catalog scope already does; more land with each plan), extend the hook to run each registered scope. This turns "tests always run through the pinned toolchain, never a host install" from documentation into an enforced invariant for AI-driven work too.

SETUP9 · Weekly budget rhythm (2 minutes, Mondays)

  • Check both weekly meters (all-models + Sonnet-only) at claude.ai → Settings → Usage.
  • Premium cap healthy → schedule an ultraplan or a heavy planning/review block this week. Premium cap tight → it's an execution week: Sonnet executors grinding waves.
  • Launch the biggest execute wave right after a 5-hour window reset, not at the end of one.
  • Sustained ceiling-hitting during heavy phases (3, 6, 9) is the actual signal to move 5x → 20x — not vibes.

Deliberately skipped
