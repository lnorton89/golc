## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| committed toolchain metadata to archive acquisition | Go/Node executable bytes are trusted only after exact platform selection, official-origin validation, and committed SHA-256 verification. |
| pull-request content to GitHub-hosted workflow | Untrusted repository changes can execute in hosted runners, but the observation job must have read-only repository permission, no secrets, and no mutation route. |
| ambient Mage launcher to repository Magefile | The exact official Mage module is installed by hosted Go only to load repository targets; project bootstrap remains the authority for project-local tool installations. |
| workflow environment to bootstrap options | A single string controls whether pinned Node/npm tooling is installed; malformed values must fail before effects and must never redirect to host Node. |
| tools-update proposal to strict platform authority | One proposal could otherwise produce mixed versions/platforms or drop a checksum while rewriting the committed manifest. |
| MCP client to Mage metadata | External clients may inspect option/target metadata but must never gain an execution path through the read-only server. |
