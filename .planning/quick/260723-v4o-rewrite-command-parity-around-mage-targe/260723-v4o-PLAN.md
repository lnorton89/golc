**Locked parity sequence:** The committed `commands.pr.steps` value remains `bootstrap,generate --check,check --offline,build,test,package --foundation`. The workflow expresses that sequence with exactly these Mage targets:

| Order | Workflow line | Shared descriptor resolves to |
|-------|---------------|-------------------------------|
| 1 | `run: mage Bootstrap` | bootstrap |
| 2 | `run: mage GenerateCheck` | generate --check |
| 3 | `run: mage CheckOffline` | check --offline |
| 4 | `run: mage Build` | build |
| 5 | `run: mage Test` | test |
| 6 | `run: mage PackageFoundation` | package --foundation |

Do not create a Go map or slice containing this six-row mapping. The table is an acceptance contract only: production logic loads the expected side from `delivery.LoadPRGraph` and resolves the actual side from `delivery.LookupMageTarget`/the defensive `MageTargets` descriptors. A route-kind descriptor contributes its declared Route/Args; the bootstrap-kind descriptor contributes its registered target name as the bootstrap graph route; the PR-kind descriptor is not a legal workflow step because it would recursively execute the whole graph.

**Workflow scan contract:** Every YAML `run:` directive in `check.yml` must be a single-line `mage <Target>` invocation with no shell operator, prefix, suffix, or argument. This closed-world rule prevents an extra curl/download/Linear/shell command from escaping sequence comparison. Independently preserve the whole-file forbidden-trigger scan, required pull_request scan, secret-name scan, and Linear/apply-capable mutation token scan. `delivery.LoadPRGraph` remains the authority for `commands.pr.network_steps` and `commands.pr.mutation_steps`; parity must fail closed on loader errors and may not reparse those config strings.
