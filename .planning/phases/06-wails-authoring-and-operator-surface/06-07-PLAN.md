<phase_goal>
**As a** playback operator, **I want to** run a prepared show through on-screen, keyboard, and constrained generic MIDI controls with independent local safety actions, **so that** I can operate reliably without learning the authoring system and without the frontend ever owning playback.

This slice makes the constrained operator surface authorable on screen (PLAY-03): in-place assignment toggles in the authoring view (D-01), multiple named surfaces (D-02), individual-item granularity (D-03), and visible-but-locked rendering (D-04) enforced server-side. After this plan an author can build a real handoff surface and see exactly what an operator can and cannot touch.
</phase_goal>

<objective>
Fill the `SurfaceService` binding over the `operatorsurface` command layer (06-01) and build the operator-surface UI: a surface list for multiple named surfaces (D-02), in-place per-item assignment toggles in the authoring view (D-01/D-03), and a visible-but-locked renderer (D-04) whose lock is enforced server-side via `operatorsurface.Authorize`, not by hiding alone.

Purpose: PLAY-03 is the "hand a simple controller surface to another person" core value made concrete. The UI is the authoring half; the server-side Authorize (06-01) is the enforcement half — both are required so "constrained" is real, not cosmetic.

Output: `internal/wails/svc_surface.go` + `OperatorSurface`, `AssignmentToggle`, `SurfaceList` components.
</objective>
