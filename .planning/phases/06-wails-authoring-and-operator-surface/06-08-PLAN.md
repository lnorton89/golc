<phase_goal>
**As a** playback operator, **I want to** run a prepared show through on-screen, keyboard, and constrained generic MIDI controls with independent local safety actions, **so that** I can operate reliably without learning the authoring system and without the frontend ever owning playback.

This final slice makes generic MIDI real: a CGo-free device driver, per-control learn with conflict rejection (PLAY-04/D-05/D-06/D-07/D-08), and cross-to-catch soft-takeover sliders with a live-position + ghost-marker visual (PLAY-05/D-09..D-12). After this plan an author can learn a controller onto a surface and take over faders without value jumps.
</phase_goal>

<objective>
Add the live MIDI device layer (`internal/midi/driver.go`, gomidi + midicatdrv, CGo-free) and wire it to the operator surface: fill `MidiService` for per-control learn (using 06-03's conflict-checked `ProposeMapping`, persisting per-surface via 06-01's `AddMidiMapping`) and soft-takeover arbitration (06-03's `TakeoverState`, unthrottled decision + throttled visual push via 06-05's events), and build the MIDI panel UI — per-control Learn affordance (D-05), the mapping list, and the live-position + ghost-marker soft-takeover slider (D-09/D-10).

Purpose: PLAY-04/05 complete the operator's third control surface. The pure logic (06-03) and per-surface storage (06-01) already exist and are tested; this plan supplies the device driver and the UI, gated by the package-legitimacy checkpoints RESEARCH.md's audit requires. Real per-device acceptance stays gated by the open MIDI-HW-02 blocker.

Output: `internal/midi/driver.go` + `internal/wails/svc_midi.go` + `MidiPanel`, `MidiLearn`, `SoftTakeoverSlider`.
</objective>
