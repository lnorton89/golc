<phase_goal>
**As a** playback operator, **I want to** run a prepared show through on-screen, keyboard, and constrained generic MIDI controls with independent local safety actions, **so that** I can operate reliably without learning the authoring system and without the frontend ever owning playback.

This slice makes the safety cluster and live status visible and operable on screen: the persistent hold-to-confirm emergency panel (PLAY-06/08/09) and the always-on status readout (PLAY-07). After this plan a user can see live show state and hit Blackout on screen, both routed to the same daemon override state as the hotkeys.
</phase_goal>

<objective>
Fill the `SafetyService` binding and the throttled status push, and build the two persistent chrome components: the fixed-position hold-to-confirm safety cluster (D-13/D-14/D-15) and the PLAY-07 live status bar (active scene, layers, BPM/bar, controlling source, output state, with an explicit idle/stopped state). On-screen safety buttons forward to the same daemon safety routes (06-02) the OS-level hotkeys use — a second independent trigger into the one override state.

Purpose: PLAY-07 is the operator's window into "who is controlling and what is the output," and the safety cluster is the on-screen half of PLAY-06/08/09. Both are persistent chrome the whole app depends on.

Output: `internal/wails/svc_safety.go` + `events.go` (status push) + `SafetyCluster` and `LiveStatusBar` components.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| webview -> SafetyService binding | On-screen safety intent crosses into daemon override dispatch |
| daemon -> webview status push | Throttled telemetry crosses into the (non-authoritative) UI cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-12 | Spoofing / accidental action | hold-to-confirm safety controls | medium | mitigate | Hold-to-confirm (D-14) guards against a stray misclick triggering an emergency action; early release cancels. |
| T-06-13 | Information disclosure / stale state | live status bar as source of truth | medium | mitigate | Status bar renders throttled hints and re-queries authoritative `FetchStatus` on a gap; never treated as the playback truth (Pitfall 1). |
| T-06-14 | Denial of Service | status push cadence | low | accept | Push is fixed-cadence and decoupled from the engine tick; a slow webview cannot backpressure the engine. Rationale: emit is throttled, non-blocking. |
</threat_model>
