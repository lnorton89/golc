<objective>
Close VERIFICATION.md Gap B[0] for PLAY-10: build genuine on-screen fixture-patch UI. Currently fixture patching is CLI-only — no `frontend/src/components` surface exists for assigning pool fixtures to concrete modes/deployments. This plan adds a `FixturePatchService` Go binding and a `FixturePatch` frontend component that drive the already-existing, already-tested `pool`/`deployment` command routes (`internal/command/pool.go`, `internal/command/deployment.go`) — a UI-binding exercise against stable backends, architecturally identical to how 06-07's SurfaceService/OperatorSurface wired UI onto the pre-existing operatorsurface routes.

Purpose: satisfy ROADMAP Phase 6 Success Criterion 1's "fixture ... workflows through on-screen controls" clause and the new PLAY-10 requirement.
Output: a working end-to-end fixture-patch slice (on-screen control -> existing command route -> real ShowState read/write).

## Phase Goal (MVP vertical slice)

**As a** show author, **I want to** patch fixtures — creating pools, assigning fixtures to concrete modes with impact review, and activating deployments — through on-screen controls, **so that** I can adapt a fixture pool to a deployment without dropping to the CLI.

Flagged assumptions (Claude's Discretion — no CONTEXT decision covers PLAY-10; recorded, not silent):
- **Interaction pattern:** a table/list surface (pools with their members; deployments with active state), NOT a wizard — consistent with the existing dense-console components (OperatorSurface/MidiPanel) and the Core Value ("adapt fixture pools ... in one or two actions").
- **Scope boundary:** the existing routes assign fixtures to a pool at a `mode` (`pool update --add "<stable_key>|<content_hash>|<mode>"`) and manage deployment lifecycle (`deployment create`/`activate`). No existing command route sets a per-instance universe/address, so per-instance universe/address editing is out of scope for this UI-binding round — this plan binds what the backend exposes and does not invent new command semantics.
- **Fixture selection:** the add-member control sources selectable fixtures/modes from whatever read the backend provides (`show inspect`, and `internal/command/fixture.go` if it exposes a list/inspect read); if no structured fixture-list read exists, expose a validated `<stable_key>|<content_hash>|<mode>` entry whose values come from `fixture inspect` output. Confirm the available read while reading fixture.go in Task 2.
</objective>
