<phase_goal>
**As a** playback operator, **I want to** run a prepared show through on-screen, keyboard, and constrained generic MIDI controls with independent local safety actions, **so that** I can operate reliably without learning the authoring system and without the frontend ever owning playback.

This plan delivers the thinnest working end-to-end Wails slice: the desktop app launches, attaches to the daemon (spawning it if needed), renders a persistent global layout, and — critically — registers the OS-level safety hotkeys that fire even if the webview is hung. After this plan a real user can launch the GUI and trigger Blackout by keyboard through the OS-level path.
</phase_goal>

<objective>
Scaffold the entire Wails v2 GUI layer (greenfield — no frontend/wails.json exists yet): the Go host (`internal/wails`), a `cmd/golc-desktop` entrypoint, the React shell, per-feature Go binding service stubs, the throttled event scaffold, and — the phase's hardest constraint — the OS-level safety-cluster hotkeys (D-16) registered by the Go host via `golang.design/x/hotkey`, calling the daemon safety routes (06-02) directly. The plan establishes the binding-aggregation and component-stub structure so the Wave 3 feature plans fill disjoint files and run in parallel.

Purpose: Every remaining GUI slice mounts into this shell and fills one pre-registered binding service + one pre-stubbed component. Getting the hotkey/daemon-supervision spine right first de-risks RESEARCH.md Assumption A3 (hotkey vs. Wails message-loop conflict) at the earliest point.

Output: Wails scaffold + `internal/wails/{app,hotkey,events,svc_*}.go` + `cmd/golc-desktop/main.go` + `frontend/` shell with region-component stubs.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| new Go modules -> build | Newly-introduced hotkey/Wails modules cross into the trusted build (T-06-SC) |
| OS keyboard -> Go host hotkey callback | Global keystrokes cross into safety-route dispatch |
| Wails host -> spawned daemon child | Supervised subprocess boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-SC | Tampering | go get of golang.design/x/hotkey ([ASSUMED]) + Wails | high | mitigate | Blocking human checkpoint verifies module legitimacy/versions and CGo-free status before install; `go mod verify` after. |
| T-06-10 | Denial of Service | hotkey.Register for the 3 safety keys | high | mitigate | Check every Register return; a failed registration surfaces a visible error, never silent (Task 1). |
| T-06-11 | Elevation of Privilege | safety hotkey callback path | high | mitigate | Callback forwards `artnet safety --source manual` directly from the Go host via ipc.Forward, never a JS-mediated path a hung/hostile webview could intercept (Pitfall 1). |
</threat_model>
