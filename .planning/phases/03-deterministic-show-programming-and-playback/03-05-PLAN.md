## Phase Goal

**As a** show author, **I want to** record, update, rename, reorder, duplicate, and delete my programming objects with undo and redo, **so that** I can iterate on a show freely and reverse mistakes.

This slice (03-05) completes the PROG-07 CRUD surface and adds the session-only linear undo/redo mechanism that the future UI (Phase 6) session will drive; it depends on every programming/scene object type existing (03-02/03/04).

<objective>
Deliver the full record/update/rename/reorder/duplicate/delete CLI surface for programming objects plus the single whole-session linear undo/redo history (PROG-07, D-12/D-13/D-14). History is an in-memory library, never persisted; the CRUD verbs mutate existing `show.State` fields (no new field is added).

Purpose: PROG-07 is the last programming requirement; it makes the object library editable and reversible. Keeping history session-only (D-14) preserves Phase 5's rule that the saved show — not the edit log — is the durable unit.
Output: `internal/programming/history.go`, additional CRUD routes in `internal/command/programming.go`.
</objective>

    - Undo then Redo returns to the identical state (round-trip idempotency).
    - Undo on an empty history returns GOLC_HISTORY_NOTHING_TO_UNDO (no crash); Redo with no redo tail returns GOLC_HISTORY_NOTHING_TO_REDO.
    - One global stack covers mixed object types (a theme edit, then a chase edit, then a scene edit undo in reverse insertion order) — assert a single ordered walk, not per-type stacks.
    - History holds no reference that would be serialized: it is a plain in-memory structure with no show.Save path.
  </behavior>

Do NOT add any persistence: history is never a `show.State` field and never touches `show.Save` (D-14).
