<objective>
Close Gap 2 / CR-02 (Critical, blocks LINR-04 and roadmap SC5 second clause): `tools/linear-sync/src/adapter.ts`'s `readOperation` (lines 198-204) is the only SDK call site in the file with no `try`/`catch`, unlike `createOperation`, `updateOperation`, and `confirmReadback` (which calls the identical `readByEntity` helper and catches). An SDK read failure — the documented behavior for a missing/archived/deleted remote object — propagates uncaught out of `LinearSdkAdapter.execute` into `cli.ts`'s unwrapped `handleLine`, permanently stopping the NDJSON `for await` reader loop for the rest of that Node process's lifetime. Because `internal/trace/transport/process.go`'s `ProcessClient` is one long-lived process reused across every `Call()` in a run, every subsequent remote operation then hangs until `GOLC_LINEAR_SYNC_TIMEOUT_MS` (default 30s) fires — turning one archived/deleted Linear object into a repeated 30-second-timeout storm instead of the graceful `{found:false}` outcome `ReadResult` exists to express.

Purpose: Make the LINR-04 truth ("reports ambiguity, partial GraphQL errors, pagination, and rate limiting without blocking ...") hold on the adjacent read-exception path too, matching the safe reported-outcome discipline every sibling call site already uses.

Output: `readOperation` wrapped in try/catch returning `{found:false}`; a new offline unit test proving a throwing read resolves to `{found:false}` (this exception path currently has zero coverage anywhere in the suite); a Go-side acceptance scenario documenting the end-to-end recovery.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Go ProcessClient <-> long-lived Node adapter (NDJSON) | A single Node process is reused across every Call() in a run; a read exception in one operation can corrupt the reader loop for all subsequent operations. |
| Node adapter -> Linear GraphQL SDK | An SDK single-entity read (project/projectMilestone/issue) can throw for a missing/archived/deleted remote object. |

Verified by the Task-1 throwing-read unit test and the Task-3 acceptance scenario. |
