- `tools/linear-sync/src/adapter.ts` maps that contract onto the official `@linear/sdk` `LinearClient` with zero reconciliation policy: exhaustive `switch`-over-`EntityKind` dispatch (compile-time `never`-checked), and every create/update calls back through `readByEntity` before returning, so a mutation's own payload echo is never trusted.

- `internal/command/linear_sync.go` (new) self-registers the `linear-sdk` build scope and `linear-sdk-operations` quick-test scope through the pre-existing Plan 17 `MustDeclareRoute`/`MustDeclareScope` contract; `config/commands.toml` documents both scope names/directory in prose (see Decisions for why not as new TOML keys).
- `internal/bootstrap/cache.go`'s `ProjectCacheLayout`/`OfflineEnvironment` gain `NpmCache`/`NPM_CONFIG_CACHE`, extending the existing Go cache-layout contract to cover the new Node/npm cache directory bootstrap now warms.
- `internal/projectconfig/model.go`'s strict "toolchain" concern `Spec` gains the five `toolchain.node.*` keys (mirroring the five existing `toolchain.go.*` keys) so the new `config/toolchain.toml` section passes strict single-authority validation.

## Decisions Made

- **Node 24.18.0's SHA-256 pin was fetched live from the official `nodejs.org/dist/v24.18.0/SHASUMS256.txt`** (not invented or approximated), matching the exact official-source-checksum discipline `config/toolchain.toml` already applies to the Go pin.
- **`package-lock.json` was generated once via host npm** (network access confirmed available in this environment) rather than hand-authored — a real npm-resolved lockfile is the only reliable way to get exact transitive-dependency integrity right (`@graphql-typed-document-node/core`, `graphql`, and TypeScript's per-platform optional binaries). This is a one-time authoring step outside `bootstrap`'s scope, mirroring how `go.sum` is committed once and only ever verified (never regenerated) by later bootstraps.
- **`internal/projectconfig/model.go`'s strict Spec was extended** with `toolchain.node.*` keys — required so the new `config/toolchain.toml` section doesn't fail strict `GOLC_CONFIG_UNKNOWN_KEY` validation. `model.go` was not in this plan's declared `files_modified`, but this was an unavoidable Rule 3 (blocking) consequence of the file that *is* declared (`config/toolchain.toml`).
- **`build.go`'s `--scope` dispatch resolves the pinned Node/TypeScript compiler at request time** (`resolvePinnedNodeExecutable(request.Root)`), not through test.go's pre-existing static-`Command` `NodeScopeRegistration` mechanism, because that mechanism bakes its executable path at package-init time (before any `Request.Root` is known) and was not designed for the correctness guarantee a build compiler path needs.
- **`config/commands.toml` documents `linear-sdk`/`linear-sdk-operations` in prose, not new TOML keys** — `internal/projectconfig`'s strict "commands" concern `Spec` is a closed six-key whitelist (D-09); adding keys there would require the same kind of Spec change already made for `toolchain.node.*`, and the actual registration authority for these two scopes is Go-side self-registration (`internal/command/linear_sync.go`), consistent with the precedent Plan 01-14's own frontmatter sets for its own future `linear-transport-pagination` scope ("registers exact Node scope ... through the Plan 17 registry contract").
- **`linear-sdk-operations`'s registered `Command` resolves the pinned Node executable via `GOLC_PROJECT_ROOT`** (the environment variable `golc.ps1` already exports before delegating every non-bootstrap route) rather than `request.Root`, because `MustDeclareNodeScope`'s `Command []string` is fixed at package-init time. If resolution fails (e.g. Node not yet bootstrapped, or invoked without the shim in a unit test), it falls back to a clearly-named placeholder executable rather than panicking — a package-init panic would crash every route in the CLI over one unbootstrapped scope, not just this one.
