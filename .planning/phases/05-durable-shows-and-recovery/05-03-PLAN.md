<objective>
Build the storage-layer engine for safe schema migration (SHOW-05): a verified backup (`VACUUM INTO` + read-back-and-validate, D-09), an ordered blob-shape migration function registry (not a SQL migration framework — the store is one blob, D-02/D-03), a transactional migrate-in-temp-copy, and an atomic Windows-safe replace that never touches the original until full success (D-08). This is a headless, fully-testable storage slice; the CLI confirm flow that drives it is Plan 04.

Purpose: Make future format evolution safe-by-construction — the original file survives any interruption, and a newer-than-supported file is refused, never rewritten.
Output: `internal/show/backup.go` (+ test) and `internal/show/migrate.go` (+ test) with the verified-backup, migration-registry, and atomic-replace mechanics and their fault-injection tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| disk -> internal/show | The `schema_version` integer and blob read from an opened `.golc` are untrusted; the version drives migration-registry lookup. |
| working file -> temp copy -> atomic swap | Migration must never leave the working file half-written. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-02 | Tampering | migration registry indexed by on-disk `schema_version` | medium | mitigate | Bounds-check `schema_version` against [1, SchemaVersion] before any registry lookup; out-of-range -> GOLC_SHOW_STATE_INVALID, never a raw index. |
| T-05-03 | Tampering / Denial of Service | interrupted migration | high | mitigate | Backup-then-migrate-temp-then-atomic-swap: the original is untouched until the re-validated migrated copy replaces it via os.Rename; proven by the force-kill test, not just claimed. |
| T-05-01 | Tampering | migrated/backup blob re-read | high | mitigate | Every migrated and backed-up blob is DecodeStrict + validate before it is trusted or swapped in (T-02-10). |
| T-05-04 | Tampering / Information Disclosure | backup + temp-copy paths | medium | mitigate | Reuse `resolvePath` for backup and temp paths; no second path-resolution rule. |
</threat_model>
