## Phase Goal

**As a** show author, **I want to** validate fixtures and adapt logical pools to concrete deployments through atomic impact review, **so that** one show adapts to different rigs without being rebuilt.

**This plan's slice:** A show author can run `golc fixture import --ofl <manufacturer>/<key>` (live fetch + cache) or `golc fixture import --ofl-file <path>` (local) to bring an Open Fixture Library definition through GOLC's canonical normalization, validation, and pinning — with any lossy/unsupported constructs surfaced as explicit warnings, and once pinned the show is fully usable offline (D-07).

<objective>
Add the OFL import path (FIXT-03): parse OFL's own JSON shape into an intermediate representation, normalize it onto the canonical `internal/fixture.FixtureDefinition` (never letting OFL's channel/mode shape leak downstream — D-08), run the same strict validation + Pin pipeline 02-01/02-02 established, and record every unrepresentable OFL construct as an explicit `LossyImportWarning` on provenance (FIXT-06, D-06). Live fetch is a narrow, user-triggered GET against open-fixture-library.org or a user-configured mirror, cached locally so the show is offline-usable after pin (D-07).

Purpose: OFL is the mature, MIT-licensed fixture corpus that seeds a first-user catalog; importing through the same canonical path guarantees imported and hand-authored fixtures are indistinguishable downstream.
Output: `internal/fixture/ofl/{model,normalize,fetch}.go`, tests, a pinned local OFL corpus for offline tests, and the `fixture import` route.

**OFL API coverage note:** Per D-07, GOLC's OFL integration is deliberately narrow — fetch a single fixture's canonical JSON by manufacturer/fixture key, then cache and pin. It does NOT integrate OFL's search, manufacturer-listing, or fixture-editor/discovery surface (no show-authoring use case needs them in v1). This bounded scope is the intended integration; the search/discovery endpoints are out of scope, not overlooked.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OFL JSON (network or mirror) -> normalizer | Untrusted, possibly-malformed fixture JSON crosses into the canonical model |
| user-supplied mirror URL -> net/http | Untrusted URL could target an internal/unintended host (SSRF) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-05 | Tampering | internal/fixture/ofl/normalize.go | high | mitigate | Route all fetched/loaded OFL JSON through the same strict decode + canonical-model validation + Pin pipeline as hand-authored YAML; never trust fetched JSON to be well-formed (malformed -> GOLC_FIXTURE_OFL_INVALID). |
| T-02-06 | Tampering / Information Disclosure | internal/fixture/ofl/fetch.go | high | mitigate | SSRF guard: http/https scheme allowlist, explicit --allow-mirror opt-in for any non-default host, context request timeout, and a bounded response-size cap before the body is read/cached. |
| T-02-07 | Tampering | OFL local cache | medium | mitigate | Cache is content-addressed and the imported fixture is Pinned (FIXT-05) so a later cache change is detectable; offline-after-pin means live fetch is never on the show-usable path (D-07). |
</threat_model>
