## Phase Goal

**As a** show author, **I want to** validate fixtures and adapt logical pools to concrete deployments through atomic impact review, **so that** one show adapts to different rigs without being rebuilt.

**This plan's slice:** A show author can run `golc fixture validate <file.yaml>` and get either a validated canonical fixture summary or an actionable diagnostic — the walking skeleton of the fixture-catalog thread. This is headless CLI/domain only (D-01); no UI (that is Phase 6).

<objective>
Establish the canonical, capability-based `FixtureDefinition` model (D-08: GDTF-friendly, not hard-wired to OFL's channel/mode shape), generate its versioned JSON Schema through the existing `internal/contracts` registry (FIXT-01), decode hand-authored YAML with strict duplicate-key/unknown-field/range/semantics rejection and actionable diagnostics (FIXT-02), and expose it as the `golc fixture validate <file>` CLI route (FIXT-04, D-02 validate-only — no scaffold/generator command; D-03 "share" = the portable YAML file itself).

Purpose: This is the substrate the entire phase normalizes into. Both the OFL-import path (02-03) and the pool/deployment thread (02-04+) depend on this canonical model existing first.
Output: `internal/fixture/{model,decode}.go`, `internal/contracts/fixture.go`, generated `schemas/fixture.schema.json`, `internal/command/fixture.go`, tests, and the `go.mod` promotion of `go.yaml.in/yaml/v4` to a direct dependency.
</objective>
